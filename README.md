# Deep Wind Model Dispatch

This repository studies a hybrid wind farm with energy storage.

In plain English, the project has two connected goals:

1. Predict wind power from weather and wind-speed data.
2. Decide when storage should charge, hold, or discharge so the wind farm earns more value from electricity prices.

The original research code comes from Zach Lawrence and Jessica Yao. This fork also contains reproduction work and new dispatch experiments using Gurobi mixed-integer optimization.

## Current Status

As of June 10, 2026, this fork contains:

- Power-model training and evaluation code.
- The paper COVE-NN reproduction notebook.
- Gurobi/MIP dispatch benchmarks.
- Chronological COVE-DV training experiments.
- Corrected rolling-horizon Gurobi results using PNNL CAES storage parameters.

Important latest result:

| Method | Scope | Baseload COVE | Dispatch COVE | Improvement |
| --- | --- | ---: | ---: | ---: |
| Paper COVE-NN reference | Reported by Zach/Jessica | - | - | 32.3% |
| Corrected rolling-horizon Gurobi MIP | Full 1980-2023 dataset | 1.743062 | 1.179495 | 32.33% |

The corrected Gurobi result uses:

- Storage type: CAES
- Storage power rating: 100 MW
- Storage duration: 24 hours
- Storage energy capacity: 2,400 MWh
- PNNL CAES round-trip efficiency: 55%
- PNNL CAES depth of discharge: 80%
- Minimum state of charge: 480 MWh
- Initial state of charge: 1,440 MWh
- Grid export limit: 249 MW
- Rolling lookahead horizon: 168 hours
- Executed step size: 24 hours
- Terminal rule inside each lookahead: `SoC_final = SoC_initial`
- Chronological carryover between windows: enabled

The independent constraint check found no real physical violations. The largest nonzero values are around `1e-12`, which is numerical rounding from optimization.

### `power_model/`

This part learns the wind-power model.

In plain English:

```text
weather / wind-speed data -> neural network -> predicted wind power
```

Useful files:

- `power_model/src/main.py`: trains the neural network power model.
- `power_model/src/model.py`: neural network architecture.

This part decides how the wind farm and storage system should dispatch power.

In plain English:

```text
wind generation + electricity price + battery state -> dispatch decision
```

Useful files:

- `strategy_model/src/storage.py`: storage logic.

Useful files:

- `milp_teacher_dispatch.py`: earlier SciPy/MILP teacher experiment.

Important tracked result files:

## Main Constraints In The Corrected Gurobi Model

The corrected Gurobi run implements these constraints:

| Constraint | Equation |
| --- | --- |
| Storage capacity | `C_min <= SoC(t) <= C_max` |
| Charge limit | `0 <= P_ch(t) <= P_ES * u(t)` |
| Discharge limit | `0 <= P_dis(t) <= P_ES * (1 - u(t))` |
| Binary operating mode | `u(t) in {0, 1}` |
| Available energy | `P_dis(t) / RTE <= SoC(t) - C_min` |
| Wind-only charging | `P_dir(t) + P_ch(t) <= P_gen(t)` |
| Delivered power | `P_delivered(t) = P_dir(t) + P_dis(t)` |
| Grid export limit | `0 <= P_delivered(t) <= P_rated,grid` |
| State-of-charge update | `SoC(t+1) = SoC(t) + P_ch(t) - P_dis(t)/RTE` |
| Initial SoC | First window starts at midpoint SoC; later windows start from the previous executed SoC |
| End-of-horizon condition | `SoC_final = SoC_initial` inside each lookahead |
| Chronological carryover | Next window `SoC_initial = previous executed SoC_final` |

## Setup

Create and activate a virtual environment from the repo root:

```bash
python3.11 -m venv venv
source ./venv/bin/activate
pip install -r requirements.txt
```

For the Gurobi experiments, you also need:

- A working Gurobi license.
- The `gurobipy` Python package.

Install it with:

```bash
pip install gurobipy
```

## Reproduce Key Results

### Train the power model

Run from the repo root:

```bash
cd power_model/src
../../venv/bin/python main.py
```

The script writes run outputs under `power_model/test/`.

### Reproduce the Palouse power-model table

Run from the repo root:

```bash
cd power_model/evaluation
../../venv/bin/python reproduce_table_vii.py
```

### Reproduce the paper COVE-NN result

Open and run:

```text
strategy_model/src/reproduce_dispatch_results.ipynb
```

Expected overall result:

```text
Baseload COVE: about 102.388
COVE-NN COVE: about 69.320
Improvement: about 32.3%
```

### Run the corrected rolling-horizon Gurobi dispatch

Run from the repo root:

```bash
LC_ALL=C MPLCONFIGDIR=/private/tmp/mpl ./venv/bin/python strategy_model/optimization/rolling_horizon_gurobi_dispatch.py \
  --horizon-hours 168 \
  --step-hours 24 \
  --storage-type caes \
  --storage-rating 100 \
  --storage-duration 24 \
  --min-soc-frac 0.2 \
  --max-soc-frac 1.0 \
  --initial-soc 1440 \
  --out-dir strategy_model/optimization/rolling_horizon_gurobi_results/full_dataset_caes_100mw_24h_dod80_mid_soc \
  --progress-every 1000
```

Expected corrected full-dataset result:

```text
Baseload COVE: 1.743062
Gurobi COVE: 1.179495
Improvement over baseload: 32.33%
```

## Notes On Large Files

The full Gurobi label CSVs are large, and GitHub rejects files over 100 MB. For that reason, generated label and checkpoint CSVs are ignored by `.gitignore`.

The repo keeps:

- Scripts
- Summary CSV/JSON files
- Figures
- Smaller model files
- Reproduction notebooks

The repo does not track the largest generated full-output label dumps.

## Current Research Interpretation

The strongest current result is not that the neural COVE-DV student beats COVE-NN.

The strongest current result is:

> A chronological rolling-horizon Gurobi/MIP model using PNNL CAES parameters and explicit physical constraints matches the published COVE-NN improvement on the full 1980-2023 dataset.

COVE-DV remains a follow-up direction. The current COVE-DV student learned useful dispatch behavior, but it has not yet beaten the corrected Gurobi benchmark or the published COVE-NN result.
